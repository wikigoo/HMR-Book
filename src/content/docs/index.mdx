## 1. Executive Summary & Core Identity

**HMR (همر)** is an AI-powered **mobile-hardware advisor chatbot** engineered specifically for the
Persian-speaking Iranian market. It bridges the gap between complex consumer electronics and everyday
users by providing clear, data-driven guidance on buying, diagnosing, and maintaining mobile devices.

A foundational tenet of the project is its operating philosophy: **HMR acts strictly as a technical
advisor, not a final decision-maker.** It empowers users with recommendations and evidence while
leaving every purchasing or repair decision entirely in their hands. The service is delivered across
a **dual-platform ecosystem** — a responsive web application and a native Android app — backed by a
self-hosted AI orchestration layer.

| | |
|---|---|
| **Name** | HMR (همر) |
| **Nature** | AI chatbot specialized in mobile technology |
| **Role** | Advisor — never a decision-maker |
| **Platforms** | Web application + Android application (Flutter) |
| **Audience** | Persian-speaking users (Iranian market) |
| **Language rule** | End-user output in **Persian** (RTL, Jalali calendar); docs & code in English |
| **Honesty rule** | Never fabricate specs or prices — **live sources only** |

## 3. Operating Principles (Non-Negotiable)

Every answer the chatbot produces — and every feature the team ships — must respect four global rules:

<Steps>

1. **Persian output.** All end-user communication is in Persian (Farsi), right-to-left, with
   Jalali (Shamsi) dates. Internal manuals and code remain in English.

2. **Honesty boundary.** The AI never fabricates specifications, benchmark numbers, or prices.
   If reliable data is unavailable, it says so.

3. **Live-price-only.** Prices are quoted exclusively from live search results at answer time —
   never from model memory. Every price is accompanied by a verify-at-point-of-sale disclaimer.

4. **Advisory, not decisional.** The bot recommends, compares, and explains — the user decides.

</Steps>

## 4. Technical Architecture & System Components

The project uses a decoupled, modern **Jamstack + micro-agent** architecture designed for
performance, scalability, sanctions-resilience, and ease of maintenance. Every layer is either
globally distributed (Cloudflare) or fully containerized and reproducible from version control —
so no single vendor can lock the project in.

| Component | Technology | Specifications & Configuration |
| :--- | :--- | :--- |
| **AI Chatbot Engine** | **FlowiseAI** | Agentflows v2 / Multi-Agents, self-hosted (Docker, `flowiseai/flowise:3.1.3`). Active chatflow: **`HMR-Agentflows-v2`**. Model routing via OpenRouter: LLM `gemini-3.1-flash-lite`, embeddings `openai/text-embedding-3-large`, reranker `google/semantic-ranker-default-004`. |
| **RAG Knowledge Layer** | **Flowise Document Stores** | Three brand-scoped stores — **Apple · Samsung · Xiaomi** — each wired to its own retriever node. Source corpora are version-controlled in the `HMR-RAG` repository. |
| **Live Web Search** | **SearXNG** | Self-hosted metasearch engine (Docker container `docker-searxng-1`) powering the chatbot's live search tool — the backbone of the *live-price-only* rule. JSON API enabled, reached internally at `http://searxng:8080`. |
| **Backend Server (VPS)** | **Docker / Ubuntu 24.04** | Containerized stack behind an nginx reverse proxy. Nuremberg · 2-core CPU · 4 GB RAM · 40 GB storage · dedicated IP. Port 3000 bound to localhost only; UFW allows 22/80/443. |
| **Web Application** | **Astro** | Modern SSG frontend, deployed globally via **Cloudflare Workers** — fast, free-tier friendly, and reachable without restriction. |
| **Android Application** | **Flutter** | Android-first build (`ir.hmrbot.app`) with a web build served at `/ai`. Glassmorphism UI, Vazirmatn typography, 100 % on-device storage (SQLite + SharedPreferences). |
| **Authentication** | **Google Sign-In** | Via Firebase / Google Cloud — Project ID `ir-hmrbot-app`, Project Number `829078792642`. Login is optional; chat works without an account. |
| **UI / UX Design** | **Claude Design** | The complete design system — tokens, components, and screens — lives in the `HMR-Design` repository. |
| **System Monitoring** | **UptimeRobot** | 24/7 uptime and health tracking across the critical endpoints. |

<Aside type="note" title="Security posture">
The FlowiseAI prediction endpoint is **private** (`isPublic=0`). The `Authorization: Bearer` token is
injected **server-side by nginx** (`/etc/nginx/hmr-auth.conf`, mode 600) and never ships in any client
bundle — neither the website nor the mobile app carries a secret. Nightly backups of the Flowise data
(`/root/.flowise`) are delivered to Telegram, and a nightly state snapshot is committed to the
`HMR-VPS` repository so the latest server state is always inspectable without VPS access.
</Aside>

## 5. How an Answer Is Produced

<Steps>

1. **A user asks a question** at [hmrbot.com/ai](https://hmrbot.com/ai/) or in the Android app —
   for example, *"گوشی میان‌رده با بهترین دوربین زیر ۲۰ میلیون؟"*

2. **The request reaches Flowise** through the nginx proxy on `srv.hmrbot.com`, which authenticates
   it server-side and forwards it to the active agentflow.

3. **The supervisor agent routes the query** — brand-specific questions consult the matching
   RAG document store (Apple / Samsung / Xiaomi); price questions trigger a live SearXNG search.

4. **The LLM composes the answer in Persian**, grounded in retrieved documents and live results,
   with the mandatory price-verification disclaimer where prices appear.

5. **The user decides.** HMR presents evidence and trade-offs — the final choice is always theirs.

</Steps>

## 8. DevOps: Deployment, Backup & Disaster Recovery

The operations model is **"edit on Windows, push to GitHub, the server updates itself"** — no manual
SSH sessions are needed for routine changes.

<Steps>

1. **Continuous deployment.** Every push to `main` in `HMR-VPS` triggers the *Deploy* workflow:
   configs are synced to the server, `nginx -t` validates the result (with automatic rollback on
   failure), and containers are recreated via `docker compose up -d`.

2. **Nightly backup to Telegram.** The *Daily Backup* workflow archives the full Flowise data
   directory every night and delivers it to a private Telegram channel — off-site, off-provider.

3. **Nightly state snapshot.** A scheduled job exports the live server state (containers, flows,
   document stores, redacted configs) into `STATE.md` and `flowise/flows/*.json` in the `HMR-VPS`
   repo, giving the team a read-only mirror of production at all times.

4. **Provider independence.** Because every config lives in git and all data fits one backup
   archive, migrating the entire backend to a new VPS provider is a **~1–2 hour documented
   procedure** — a deliberate defense against vendor lock-in and regional provider constraints.

</Steps>

## 9. Reliability & Health Monitoring

**UptimeRobot**. Monitoring evaluates three primary endpoints to verify full-stack health:

- **`hmrbot.com`** — frontend availability and Cloudflare routing.
- **`hmrbot.com/api/health`** — core API gateway and internal response status.
- **`srv.hmrbot.com`** — VPS, FlowiseAI engine, and LLM processing chain.
