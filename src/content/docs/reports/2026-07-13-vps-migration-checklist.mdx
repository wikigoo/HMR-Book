## Purpose

Move the production HMR backend (Flowise + SearXNG Docker stack and its nginx front) from the
**current** server to a **new** server, with zero data loss and a fast rollback path.

Only `srv.hmrbot.com` is served from the VPS. `hmrbot.com` and `book.hmrbot.com` run on
Cloudflare Workers (Astro) and are **not** affected by this migration.

## Authority gates (per SUPERVISOR.md)

- Every mutating command on either VPS → **explicit human confirmation**.
- DNS change in Cloudflare → **explicit human confirmation** (reversible but sensitive).
- Stopping/deleting the old server → **explicit human confirmation**.
- Changes to `HMR-VPS` → **PR only**, human merges.
- **Secrets never leave the server** — transfer server-to-server (rsync over SSH), never
  through a laptop or an agent's context.

## Phase 0 — Prerequisites

- [ ] Confirm access to: Cloudflare DNS, `HMR-VPS` repo, Telegram backup bot.
- [ ] Pick a low-traffic cutover window.

## Phase 1 — Access & hardening (new server)

- [ ] First SSH login as root to `5.75.207.85`. *(owner tests reachability)*
- [ ] `apt update && apt upgrade -y`; set timezone/hostname.
- [ ] Add SSH public key, then set `PasswordAuthentication no` in `sshd_config`.
- [ ] Configure `ufw`: allow only `22, 80, 443` — **for both IPv4 and IPv6**.
- [ ] Add local `hmrbot-server` SSH alias pointing at the new IP.

## Phase 2 — Install Docker (on the existing Ubuntu 24.04, no Rebuild)

- [ ] `curl -fsSL https://get.docker.com | sh`
- [ ] Verify versions against current server (Docker 29.5.3 / Compose 5.1.4 per STATE.md).
- [ ] `apt install -y nginx`.

> **Do not use the panel's Rebuild "Docker Registry" image** — that installs a private image
> registry, not the Docker Engine runtime HMR needs. A prebaked image also locks the box to the
> reseller's versions/config and diverges from the documented IaC. Keep the clean Ubuntu.

## Phase 3 — Back up from the current server

- [ ] 🟠 **Critical:** copy the entire `~/.flowise` directory (57M) — including
      `database.sqlite` **and the `encryption.key`**. Without the encryption key the 4 stored
      credentials (OpenRouter, etc.) become undecryptable and everything breaks.
- [ ] Identify the exact volume mapping in the current `docker-compose.yml` (host path vs named volume).
- [ ] Back up config files: `/opt/flowise/docker/docker-compose.yml`, `/opt/flowise/docker/.env`,
      `/etc/nginx/sites-available/hmrbot.conf`, `/etc/nginx/conf.d/*`, `/etc/nginx/hmr-auth.conf`.
- [ ] Take a consistent Flowise snapshot (brief container stop, or `sqlite3 .backup`).
- [ ] Document stores (Apple/Samsung/Xiaomi) live inside `~/.flowise` — no separate step.

## Phase 4 — Transfer (server-to-server, direct)

- [ ] `rsync -avz` directly old → new over SSH; secrets never pass through a third party.
- [ ] Move: full `~/.flowise/`, the three nginx config files, `docker-compose.yml`, `.env`.
- [ ] Recreate path layout: `/opt/flowise/docker/` and the data path.

## Phase 5 — Deploy the stack (new server)

- [ ] `docker compose up -d` → `docker-flowise-1` and `docker-searxng-1` come up.
- [ ] Load nginx config; `nginx -t`; `systemctl reload nginx`.
- [ ] Issue TLS cert for `srv.hmrbot.com` on the new box **before** DNS switch
      (certbot `--webroot` or DNS challenge).
- [ ] SearXNG from the new box (same Hetzner Nuremberg DC) — expect only bing/qwant/mojeek to
      work, same as current.

## Phase 6 — Verify before DNS switch (critical)

- [ ] End-to-end test without touching production, via local override:
      `curl --resolve srv.hmrbot.com:443:5.75.207.85 https://srv.hmrbot.com/...`
- [ ] Confirm containers healthy (`docker ps`), health endpoint, and a live prediction on the
      `HMR-Agentflows-v2` chatflow.
- [ ] Confirm credentials decrypt (a real model response proves the encryption key transferred).
- [ ] Test Google login (and that anonymous chat still works).

## Phase 7 — DNS cutover

- [ ] 🔴 In Cloudflare, change the `srv.hmrbot.com` **A** record from `91.107.159.48` to
      `5.75.207.85`. *(explicit owner confirmation)*
- [ ] Lower TTL beforehand (e.g. 60s) for fast propagation.
- [ ] Reminder: `hmrbot.com` and `book.hmrbot.com` are on Cloudflare Workers, **not** the VPS —
      only `srv.hmrbot.com` moves.
- [ ] Optional: add an AAAA record for IPv6 now.

## Phase 8 — Post-cutover verification

- [ ] Real chat test on `hmrbot.com/ai` (full chain: Flutter → proxy → new srv).
- [ ] Review Flowise and nginx logs for errors.
- [ ] Monitor 24–48h before decommissioning the old server.

## Phase 9 — Re-point monitoring & backups

- [ ] UptimeRobot: verify the `srv.hmrbot.com` monitor (auto-resolves new IP).
- [ ] Install the nightly backup script (archive → Telegram) and `snapshot_state.sh` on the new
      box; set the cron jobs.
- [ ] First new snapshot in `HMR-VPS` should reflect the new server.

## Phase 10 — Decommission the old server (after a grace period)

- [ ] After 48–72h of stability, take a final backup of the old server.
- [ ] 🔴 Stop/delete the old Hetzner box. *(explicit owner confirmation)*
- [ ] Update `FACTS.md` (new IP, new provider) and `STATE.md` via **PR**.

## Rollback

If anything breaks after cutover: revert the Cloudflare A record to `91.107.159.48`. Because the
old server stays untouched until Phase 10, rollback takes only a few minutes.
